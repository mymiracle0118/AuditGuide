Guidelines

writing commentary justifying every single line/logical block of code that I review
Being able to explain with my own words the protocol to a non-tech person
Have base level understanding of what the protocol actually does
Look for way users interact with the codebase (this part when you can start thinking about how to mess with certain components but you won’t find the gnarly bugs here)
Go deeper into the technical documentation (this can be tedious and boring but 100% necessary)
Break any assumptions made by developers (this probably where you spend the least amount of time but here is most effective - bugs can be found here, important!)
If you are stuck for while (X hours) have a break or ask someone helps

Audit Process ​

./audit_setup.sh ... GITHUBLINK
Scan auditbase + add report in AUDIT folder
Read the README.md
Skim through the code (4 ~ 5 hours focused ~ 500 SLOC/h)
CMD + K + CMD + 0
Get an overview idea of what the protocol should be doing and the architecture → move quickly, don’t focus on details
Make questions in code
Add first thoughts and notes (@audit / @audit-info / @audit-ok / @audit-issue)
Things that just feel off
Ways I can get the protocol to misbehave
Obvious issues
Areas to come back to go deeper later (complex math)
Skim through the tests
Run Slither slither .
Read line by line, word by word
CMD + K + CMD + J
Keep adding notes with tags (@audit / @audit-info / @audit-ok / @audit-issue)
UNDERSTAND every single line of code
Thinking about exploits
Exploring rabbit holes
Read all the documentation (before moving on, make sure I really understand the system)
Review diagrams
Review flows
Watch YouTube videos if available
Read past audit reports of this protocol if available
Read past audit reports of similar protocols if available
Ask questions (what if)
write down the "what happens if I can change X" and "can we break X invariant", etc list of questions → this gives you targeted problems to attempt to solve instead of reading line after line of code randomly looking for issues
Work Backwards from Exploits
"For X to happen, either Y or Z would need to be true." "Y can't be true because it's an invariant based on ABC." "Z could be true, but only in this weird unlikely case..."
Resolve any tags @audit
Look through Solodit for vulnerabilities and ideas in similar protocol
Run thought the checklist below
Write a PoC all relevant H/M findings
Write report
End audit when all tags have been resolved

External call

[E-01] - Is an external contract call actually needed?
[E-02] - Is the address called whitelisted ?
[E-03] - Mistrust when there is a fixed gas amount in a .call() and check if the gas left is enough
[E-04] - Grief attack is possible when calling a unknown address by passing huge amount of data into the call ⇒ use inline assembly.
[E-05] - A call to an address that do not exist return true : is the existence of the address checked ?
[E-06] - Use the check, effect, interaction pattern. Only If necessary use a reentrancyGuard but beware of cross contract reentrancy
[E-07] - For sending ETH don't use transfer() or send() and instead use call()
[E-08] - msg.value not checked can have result in unexpected behaviour
[E-09] - Delegate calls that do not interact with stateless type of contract (library) should be triple check
[E-10] - Never delegate call to an untrusted contract
[E-11] - If the recipient of ETH had a fallback function that reverts, could it cause DoS?
[E-12] - Could it cause an out-of-gas in the calling contract if it returns a massive amount of data? Can the external call be manipulated to cause DoS?
[E-13] - Would it be harmful if the call reentered into the current function?
[E-14] - Would it be harmful if the call reentered into another function?
[E-15] - What if it uses all the gas provided?
[E-16] - Could it cause an out-of-gas in the calling contract if it returns a massive amount of data?
[E-17] - Care when using msg.value in a multi call.

General tips

[G-01] - For logic implemented several times, see if there are any differences and then standardize the logic.
[G-02] - Timelocks should be implemented for every protocol upgrade/change. (to let users exit if they don’t agree)

Self-destruct
Deterministic address can be force feed before deployment
(Coinbase Transactions : The attacker can start proof-of-work mining and set the target address to receive the reward)
[G-04] - Beware of Dos

Sensible withdraw logic should be done externally by the user
Beware of Block Stuffing attack when the contract need to do an action within a certain time period.
Maybe if there is a timelock a attacker can brick the logic at no cost
[G-05] - Beware of Oracle manipulation: Don't use spot price from an AMM as an oracle.
[G-06] - When deleting a structure you should delete mapping inside first
[G-07] - Be careful of relying on the raw token balance of a contract to determine earnings
[G-08] - Take care if (receiver == caller) can have unexpected behaviour
[G-09] - When pause mechanism :
can pause something that should not be pause (e.g liquidation)
